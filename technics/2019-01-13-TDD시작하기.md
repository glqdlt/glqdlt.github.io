## TDD 란?

> Clean code that woarks (잘 동작하는 깔끔한 코드) - Ron Jeffries(론 제프리)

> Test the program before you write it(프로그램을 작성하기 전에 테스트를 먼저 작성하는 것) - Kent Beck(켄트 백)

TDD에서 말하는 테스트와 전통적인 테스트 방법론에서 이야기하는 단위 테스트는 조금 차이가 있습니다. TDD에서의 단위 테스트는 함수(기능적) 단위의 테스트를 뜻하고, 전통적인 방법론에서는 사용자 측면에서 제품의 기능 테스트에 가깝습니다. 

예를 들어 로그인을 한다는 행위를 놓고 두 방법론을 비교해본다면 전통적인 방법론에서의 단위 테스트는 '로그인 입력 창에 ID/PW 를 입력하고, 버튼을 누르고 로그인 성공 화면을 기다린다' 까지가 되며 TDD 방법론 에서는 ID 와 PW 의 Validation 을 담당하는 기능의 테스트, 버튼이 제대로 Submit 되는지의 테스트, 로그인 성공 후의 응답 테스트 등이 있을 수 있습니다.


테스트 코드는 매우 중요합니다. 임베디드 엔지니어링을 두고 설명해보면, 기본적으로 하드웨어는 제품이 동작하기 전에 사전 테스트를 수행하고 제품의 가동여부를 판단한 후 가동시킵니다. 예를 들어 컴퓨터에서 핵심 부품 인 메인보드의 경우 램 슬롯에 램이 잘 장착됬는 지, CPU는 장착되었는 지 등을 사전 테스트하고 실패시에는 비프음과 함께 컴퓨터 파워를 차단시키고 구동을 멈추게 합니다. 만약 위와 같은 테스트 검증 과정이 없다면, 의도치 않은 동작이나 또는 더 굉장하고 끔찍한 상황을  마주할 수도 있게 될테니깐요.


> "최대한 빨리 실패하기" TDD는 실패를 통해 배움을 늘려가는 기법이다. OK 조건을 사전에 정해두고 빠르게 실패를 경험하며, 조건을 등대로 삼아 실패 상황을 최대한 빨리 극복하고자 노력한다. - 테스트 주도 개발 : 고품질 쾌속개발을 위한 TDD 실처법과 도구

> 우리는 흔히 클래스를 설계할 때 속성(필드:멤버변수)에 집중해서 설계 하는 경향이 있다. ... 클래스를 정의할 때 중요한 건 '속성' 이 아니라 '동작(메소드)' 이다. 동작을 먼저 정의하고, 동작에 필요한 속성을 고려한다 는 식으로 접근하는 편이, 불필요한 속성이 클래스 내에 섞여 들어가는 걸 줄여준다. - 테스트 주도 개발 : 고품질 쾌속개발을 위한 TDD 실처법과 도구

> "테스트에서 오류와 실패" 실패는 AssertEqauls 등의 테스트 조건식을 만족시키지 못했다는 것을 의미하고, 오류는 테스트 케이스 수행 중 예상치 못한 예외(exception)가 발생해서 테스트 수행을 멈췄다는 것을 뜻한다. 테스트 케이스가 가치를 지니기 위해서는, 어떠한 경우에도 테스트 케이스 그 자체는 정상적으로 끝까지 수행되어야 한다.  - 테스트 주도 개발 : 고품질 쾌속개발을 위한 TDD 실처법과 도구

> "여러개의 실패하는 테스트 케이스의 빠른 선언" 순수한 TDD 론에서는 실패하는 테스트를 여러 개 만들어 놓고 진행하는 걸 권장하지 않는다.

## TDD 라이프사이클

<img src="http://2.bp.blogspot.com/-eMxpoJmZPwM/UvHY2BBeuHI/AAAAAAAAAwM/IvlkwTT4aGU/s1600/Picture1.gif"/>

1. 질문(ASK) == Write Failing Test == Red

    테스트의 기본 골자를 작성합니다. 해당 함수가 원하는 동작을 하는지를 체크하고 동작하기를 기대합니다.

2. 결과(Result,Response)  == Make Test Pass == Green

    테스트의 수행 결과를 확인하는 과정입니다. 성공할 수도, 실패할 수도 있습니다.

3. 정제(Refine) == Refactor == Yellow

    코드를 다듬는 과정. 실패했다면 실패한 이유를 분석하고, 성공, 실패 유무를 떠나 불필요한 것을 배제하고 모호한 것은 명확하게 하는 과정을 가집니다.

4. 반복(Repeat)

    위의 1,2,3 과정들을 한 사이클(1 cycle)로 보고, 다시 반복합니다.


## TDD 용어 정리

TDD 를 하면서 자주 접하게 되는 단어들에 대한 정리.

- 픽스처(FIXTURE)

    "일관된 테스트 실행환경"을 의미합니다. 즉, 테스트 기반이 되는 환경을 뜻 합니다.

    대부분 테스트 케이스에서 공통으로 사용 될 dummy 객체 인스턴스 등이 해당합니다.

    jUnit 에서 set up 인```@before``` 이나 tear down```@after``` 에 해당하는 메소드들을 픽스처 메소드(fixture method)라고 부릅니다.

- Stub

    Stub은 미리 준비된 응답([canned answer](https://en.wikipedia.org/wiki/Canned_response)) 혹은 정해진 대답이란 사전적인 의미를 가집니다. 예를 들자면 단어 외우기 시험과 비슷합니다. 출제 될 특정 단어들을 기재하고 실제 문제에서 해당 문제의 뜻을 작성하는 시험과 비슷하게 어떠한 것을 사전에 약속하고 질의응답을 하는 개념입니다. 
    
    (더 자세한 것은 [Test Stub 이란 무엇인가](https://medium.com/@SlackBeck/%ED%85%8C%EC%8A%A4%ED%8A%B8-%EC%8A%A4%ED%85%81-test-stub-%EC%9D%B4%EB%9E%80-%EB%AC%B4%EC%97%87%EC%9D%B8%EA%B0%80-ff9c8840c1b0) 를 참고)

    내부 메소드에 결과가 이럴 것이다 라고 기대(expected)하는 final 사수를 작성할 때, 상수 이름을 뭘로 해야할 지 고민이 됨. (https://github.com/freerange/mocha/issues/164) stub 으로 하는 게 옳다고 생각했는 데, 사실상 expected 가 더 명확한의미다 보니 이에 대한 고민이 필요한듯

- Fake


지금은 절판 되었지만, 아주 좋은 책으로 [테스트 주도 개발 : 고품질 쾌속개발을 위한 TDD 실천법과 도구]()의 186page 를 보면 내가 고민하는 내용에 대한 명쾌한 대답이 있다.

단위 테스트를 하는 입장에서 dummy, mock, stub 은 모두 비슷비슷해 보인다. 그도 그럴것이, 짝퉁이라는 개념에서는 모두 같은 의미이다. 이에 대해서 [xUnit Test Patterns]() 의 저자인 제라드 메스자로스(Gerard Meszaros)는 테스트 더블(Test Doutble, 대역 스턴트맨을 나타내는 스턴트 더블의 유사 파생어)이란 용어를 만들고, dummy, mock, stub 을 모두 하뤼 카테고리로 분류 하였다.

메스자로스의 이러한 분류 방식은 마틴 파울러가 자신의 블로그에 게시한 이후 대부분의 사람들에게 알려졌다. 그러나, 알려졌을 뿐이지 실제 많은 개발자의 손가락에는 녹아나지 못하고 아직도 나를 비롯한 여러 개발자가 테스트 더블이란 개념을 Mock 이란 이름으로 부르고 있다.

### dummy

더미 는 단순히 인스턴스를 유지하기 위한, 모조품이란 의미를 가진다. 더미는 객체 고유의 기능을 가지지 않는다. 단순히 new SomeDummy(); 라는 키워드로 인스턴스를 생성하고만 싶을 때 사용한다. 이 말은 dummy 객체의 단순한 getter/setter 를 호출하더라도 exception 이 발생할 수 있다는 의미이다. 

### stub

dummy 가 단순한 껍데기라면, stub 은 특정한 상황을 연출할 때 사용한다. 단순한 인스턴스인 dummy 에 반해 stub 은 특정 상황의 기능이나, value 를 갖고 있는 인스턴스이다. stub 이 특정 상황의 기능을 가진다는 의미를 잘 생각해보자, 이 말은 특정 기능의 세세한 로직이 선언되지 않았음을 의미한다.

```java

public integer sum(int x, int y){
    return -1;
}

```

위 메소드는 x + y 가 작성되었어야할 코드라고 생각하지만, -1 를 뱉는 상황을 연출하는 stub 이다. 

- 단지 인스턴화 할 수 있이면 더미

- 인스턴스화 된 객체가 특정 상태나 모습을 가져야 한다면 스텁

지금까지 내가 만든 fixture 에 해당하는 테스트 인스턴스 객체는 이 stub 에 해당했다.

### fake

fake 객체는 stub 과 거의 개념적으로 같지만 미묘하지만 큰 차이를 가진다.  stub 이 특정 하나의 상황만 연출한다면, fake 는 여러가지 상황을 연출해놓은 stub 이라고 보면 된다.

```java

public integer sum(int x, int y){
    if(x > y){
        return 1;
    }else{
        return -1;
    }
}

```

어거지 예시이지만 위의 경우와 차이가 생겼다. 처음 인자로 들어오는 아규먼트 x가, y 보다 클 경우 양수를 return 한다. 이 경우 첫번째 인자가 클 경우 return 이 양수에 대한 테스트를 처리할 수 있고, 작을 경우에는 음수가 return 될 것이라는 테스트를 할 수가 있다.

페이크 객체는 복잡한 로직이나, 객체 내부에서 필요로 하는 다른 외부 객체들의 동작을, 비교적 단순화하여 구현한 객체다.

### spy

스파이란 말 자체를 보면, 정상적인 A 란 인물을 가장하고 행동하면서 B라는 일을 하거나 뒤에서 몰래 공작을 펼치는 인물을 말한다. 테스트 스파이 객체도 마찬가지이다.
스파이 객체는 fake 나 stub 객체의 기능이 제대로 수행되었었는지, 얼마나 수행되었었는지 를 체크하는 등 메소드의 목적 외의 부가적인 탐구를 하고 싶을 때 사용된다.


```java

private int spyX = 0;

public integer sum(int x, int y){
    setSpyX(x);
    if(x > y){
        return 1;
    }else{
        return -1;
    }
}

private void setSpyX(int x){
    this.spyX = x;
}

public integer getSpyX(){
    return this.spyX;
}



```

stub 객체나 fake 객체의 내부를 스파이 메소드가 더럽혀버렸다. 실제로 이래도 되나? 싶을 수도 있는 데, 어차피 stub 이나 fake 객체는 test 코드 외에서는 사용될 이유가 없기에 객체 내부를 테스트만을 위한 로직 작성을 해도 무관하다.

위처럼 스파이 메소드가 내부에 있존재할 경우 sum(2,1) 메소드를 호출하고, 1 가 나왔지만 정말로 x가 sum(..) 내부로 잘 들어갔고, 들어갔을 때 값이 2인지의 여부를 getSpyX(..)를 통해 확인 할 수가 있어진다.
사실 이런 경우는 크게 많이 없고, 보통 deepCopy 의 기능을 담당하는 기능에 스파이 메소드로 원본 객체 상태의 snapshot 데이터를 내부 몰래에 만들어둘게 한 후, 실제 deepCopy 가 잘 되었는지를 체크하는 경우 등에서 사용할 수 있다.

스파이 객체는 위처럼 테스트 인스턴스 내부를 더럽혀야하는 스파이 메소드를 번거롭게 코딩해야한다는 점 때문에 잘 선호되지 않는다. 이런 이유로 Mock 제공 프레임워크(jMock,Mockito 등)에서 내부적으로 스파이 기능과 같은 것을 내장해서 개발자에게 제공해주는 추세이다.


### mock

mock 을 이해하려면 상태 기반(state base test) 테스트와 행위 기반(behavior base test) 테스트 2가지의 개념을 이해 해야한다.

- 상태 기반 테스트

객체 지향 프로그램의 특징 중 하나는 객체는 특정 시점에 자신 만의 상태를 갖을 수 있다. 상태 기반 테스트는 특정 인스턴스의 상태를 setter() 를 통해 설정하거나, stub 과 같은 특정 상태를 연출해놓고 그 상황 속에서의 기대값(expected) 을 비교해보는 식의 일반적인 테스트 기법이다.

- 행위 기반 테스트

TDD 을 넘어서자고 알려진 BDD 의 개념과는 다른 기법이다. 행위 기반 테스트는 전통적인 상태 기반 테스트의 이상의 것을 테스트하고 싶을 때 등장한다.
예를 들어 어떠한 입력이 있고, 어떠한 반응이 나올것이다는 것이 상태 기반 테스트라면, 행위 기반 테스트는 어떠한 입력이 있을 때, 어떠한 반응의 여파로 다른 반응이 존재하는가? 를 체크할 때 이다. 이게 무슨 말이냐면, 우리가 테스트 대상인 메소드의 return 타입이 void 인 경우를 얘기해보자. 대표적으로 run() 이나 startOn() 과 같은 기능의 메소드가 대표적이다. 이런 경우 테스트의 실제 호출은 run()이 되지만, 우리가 기대하는 것은 어플리케이션이 동작했거나 어떠한 다른 기능이 run 됬을 거라고 기대한다. 이런 경우를 찾아야 할 때가 행위 기반 테스트이다. 즉 행위 기반 테스트는 호출 메소드의 여파로 인해 다른 곳의 상태가 어떠한지를 알려줄 스파이 메소드가 필요해지는 경우라고도 말할 수 있다.

아까 위에서 잠시 언급했지만 spy 객체는 손이 많이 가서 잘 안사용하고, 최근엔 mock 에서 이러한 기능을 제공해주었다고 말했던 것이 기억나는가? mock 객체는 행위 기반 테스트를 검증하기 위해 사용되는 객체를 말한다. 사실 이는 제라드의 분류에서의 의미이고, 최근에는 더 넓은 의미로 '가상 임시 구현체'의 의미로 사용되는 경우가 더 많다. 이 의미는 제라드의 분류에서 가장 최상위에 있던 테스트 더블과 동일한 수준의 의미를 갖는다. 이런 뜻에서 앞서서 제라드의 분류 보다는 Mock 이라는 의미를 많이 사용한다고 언급했었던 이유이다.

그렇다면 왜 Mock 만 알면 되지, 왜 어려운 제라드의 분류학까지 들먹이면서 얘기를 했냐면 Mock 이 제라드의 최상위 분류인 테스트 더블과 동일하다는 의미는 Mock 의 쓰임새를 stub 처럼 사용하기도, 더미 객체처럼 사용할 수 있기도 하기 때문이다. 테스트 주도 개발 책에서는 이러한 Mock 의 변화무쌍한 성격 탓에 Mock 프레임워크를 사용하지 않고, 직접 명시적으로 사용하고 싶은 경우엔 익명 클래스로 Mock 객체를 만들 것을 권장한다. 상황에 맞는 클래스를 줄줄이 만들기 보다는 여러 의미를 다 담고 있다는 의미로 정의 되지 않은 빈 인터페이스를 작성하고, 필요할 때 해당 인터페이스에 추가하자는 의미이다.

## Mock 프레임워크

Mock 프레임워크는 동적으로 Mock 객체를 만들어주는 프레임워크를 지칭한다. 이점으로는 개발자가 직접 번거롭게 테스트 클래스를 정의하지 않아도 된다는 점과 행위 기반 테스트 케이스도 쉽게 사용할 수 있기 때문이다.

테스트 주도 개발 집필 당시에는 1,2 위로 easyMock 과 jMock 이 다투고 있었고, Mockito 가 이제 흥행 되고 있었다고 한다. 현재는 대세가 되어버린 Mockito가 그때에는 흥행되고 있엇다고 하니 여러 생각이 들지만, 당시 슬로건으로 'Mockito 는 간편함과 함께 기존 Mock 프레임워크들이 지향했던, 행위 기반 테스트 위주에서 상태 위주 테스트로의 회귀를 전면으로 내세웠다' 라고 한다. 
여기서 재밌는 점이 작가의 말에서 나오는 데, Mockito가 대중화 되기 이전의 Mock 프레임워크는 행위 기반 테스트 기반 프레임워크였으며 이러한 Mock 프레임워크를 자주 사용하는 개발자는 Mock 주도 개발자라 불리웠으며, 직접 행위 기반 테스트를 위한 테스트 mocking 을 만들어가며 특정 상황의 반응을 체크하던 고전주의 TDD 개발자라 불리며 서로를 야유하고 싸웠다고 한다. 이에 대해 마틴 파울러는 [mocks aren't stubs(mock은 스텁이 아니다)](https://martinfowler.com/articles/mocksArentStubs.html) 라는 말과 함께, 상태 기반 테스트만을 하기 보단 행위 기반 테스트도 필요할 때가 있다는 말로 중립적인 얘기를 했다고 한다. 


### Mockito

Mockito 는 지금에 와서 BDDMockito 가 추가로 생겼지만, 기본적으로 Mock 프레임워크 사용을 상태 기반 테스트를 해보자를 모토로 하고 있다. Mockito 가 세상에 나왔을 당시 BDD의 창시자인 댄 노스(Dan North)는 Junit4 와 Mockito 가 Java 에서 TDD와 목킹(mocking : mock 객체를 만드는 수고)의 미래다 라고 했다고 한다.

Mockito 는 수제빵 파베르(Szczepan Faber) 가 만들었다. (그 빵이 아니다) Mockito 의 지향하는 바는 'TDD를 위한 테스트 코드 작성을 간결하게 해서 개발자가 전념할 수 있도록 서포팅 한다' 이다.

테스트 코드의 템플릿을 given/when/then 식의 순서로 작성하는 패러다임은 MOckti o개발자의 한 포스팅에서 시작되었다.
